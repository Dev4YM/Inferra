# Failure Taxonomy

## Purpose

The failure taxonomy is a structured classification hierarchy for root causes. Every hypothesis generated by the hypothesis engine is tagged with a `CauseType` and `cause_subtype` from this taxonomy. This enables:

1. Consistent classification across all incidents
2. Trend analysis ("60% of incidents this week were dependency failures")
3. Structured routing of suggested checks (each taxonomy node has associated diagnostic commands)
4. Vocabulary standardization between the deterministic engine and the LLM explanation layer

## Classification Hierarchy

### 1. DEPENDENCY_FAILURE

An upstream service or external dependency fails, causing cascading errors in dependent services.

| Subtype | Description | Typical Evidence |
|---|---|---|
| `upstream_service_failure` | A service that others depend on has errors | Error on service B, then errors on services calling B |
| `upstream_timeout` | A dependency is slow, causing timeouts in callers | Timeout events on callers; high latency on dependency |
| `upstream_unavailable` | A dependency is completely unreachable | connection_refused from multiple callers to same target |
| `external_api_failure` | A third-party API is failing | HTTP 5xx from external endpoints; DNS failures |

**Diagnostic commands**:
```
docker logs <dependency_container>
curl -v <dependency_health_endpoint>
docker inspect <dependency_container> | grep State
ping <dependency_host>
```

---

### 2. RESOURCE_EXHAUSTION

The system ran out of a critical resource (CPU, memory, disk, file descriptors, connections).

| Subtype | Description | Typical Evidence |
|---|---|---|
| `memory_exhaustion` | RAM usage at or near capacity | OOM events, high memory metrics, process kills |
| `disk_exhaustion` | Disk full or near-full | ENOSPC errors, disk metrics > 95% |
| `cpu_saturation` | CPU fully utilized | Timeouts, high load average, CPU metrics > 90% |
| `fd_exhaustion` | File descriptor limit reached | "too many open files" errors |
| `connection_pool_exhaustion` | Database or HTTP connection pool depleted | "connection pool exhausted", "too many connections" |
| `thread_pool_exhaustion` | Thread/worker pool fully consumed | Request queuing, increasing latency |

**Diagnostic commands**:
```
free -h
df -h
top -bn1 | head -20
cat /proc/sys/fs/file-nr
docker stats --no-stream
ulimit -n
```

---

### 3. APPLICATION_BUG

An error originating from application code, not from infrastructure or dependencies.

| Subtype | Description | Typical Evidence |
|---|---|---|
| `crash_loop` | Application crashes and restarts repeatedly | Multiple restart events, pre-crash error logs |
| `unhandled_exception` | Uncaught exception terminating a request or process | Stack traces, "unhandled exception" messages |
| `null_reference` | Null/None pointer dereference | NullPointerException, AttributeError, TypeError |
| `assertion_failure` | Code assertion violated | AssertionError, assert messages |
| `data_format_error` | Unexpected data format (parsing failure, schema mismatch) | JSON parse errors, validation failures |
| `infinite_loop` | Code stuck in a loop, consuming resources | CPU spike on one process, timeout on requests |
| `race_condition` | Concurrency bug producing inconsistent state | Intermittent errors, state corruption |

**Diagnostic commands**:
```
docker logs <container> --tail 200
# Look for stack traces and exception messages
docker inspect <container> --format '{{.State.ExitCode}}'
```

---

### 4. INFRASTRUCTURE_FAILURE

Failure in the underlying infrastructure (network, DNS, storage, OS).

| Subtype | Description | Typical Evidence |
|---|---|---|
| `network_connectivity` | Network-level failure between services | connection_refused across multiple services |
| `dns_failure` | DNS resolution failing | NXDOMAIN, "name resolution failed" events |
| `storage_failure` | Filesystem or volume errors | I/O errors, read-only filesystem messages |
| `docker_daemon` | Docker daemon issue | Docker API errors, container management failures |
| `kernel_issue` | OS-level problem | dmesg errors, kernel panics, segfaults in system processes |
| `certificate_expired` | TLS/SSL certificate issue | certificate_error tags, SSL handshake failures |
| `port_conflict` | Port already in use | "address already in use" errors |

**Diagnostic commands**:
```
docker network ls
ping <target>
nslookup <hostname>
dmesg | tail -50
mount | grep <filesystem>
openssl s_client -connect <host>:<port>
netstat -tlnp
```

---

### 5. CONFIGURATION_ERROR

A misconfiguration in application settings, environment variables, or infrastructure configuration.

| Subtype | Description | Typical Evidence |
|---|---|---|
| `deployment` | Errors started after a deployment/release | deployment tag precedes errors |
| `config_change` | Errors started after a configuration change | config_change tag precedes errors |
| `environment_variable` | Missing or incorrect environment variable | "key not found", "invalid config" messages |
| `permission_misconfiguration` | Incorrect file/directory permissions | permission_denied errors |
| `resource_limit_misconfiguration` | Container memory/CPU limits too restrictive | OOM with low memory limit set |
| `dependency_version_mismatch` | Incompatible library or service version | Import errors, protocol errors |

**Diagnostic commands**:
```
docker inspect <container> --format '{{.Config.Env}}'
docker inspect <container> --format '{{.HostConfig.Memory}}'
git log --oneline -10  # in service repo
diff <old_config> <new_config>
```

---

### 6. DATABASE_FAILURE

Failure specifically in a database system.

| Subtype | Description | Typical Evidence |
|---|---|---|
| `connection_failure` | Cannot connect to database | connection_refused to DB port, timeout on DB connect |
| `query_failure` | Database queries failing | SQL errors, query timeout events |
| `deadlock` | Database deadlock detected | "deadlock detected" messages |
| `replication_lag` | Database replica behind primary | Stale reads, replication lag metrics |
| `schema_migration_failure` | Database migration failed | Migration error logs |
| `data_corruption` | Inconsistent data detected | Integrity constraint violations, unexpected NULL values |

**Diagnostic commands**:
```
docker logs <db_container> --tail 100
# PostgreSQL:
docker exec <db_container> psql -U postgres -c "SELECT count(*) FROM pg_stat_activity;"
docker exec <db_container> psql -U postgres -c "SELECT * FROM pg_locks WHERE NOT granted;"
# MySQL:
docker exec <db_container> mysql -e "SHOW PROCESSLIST;"
# Redis:
docker exec <redis_container> redis-cli INFO
```

---

### 7. UNKNOWN

No specific failure pattern matched. The system detected anomalous behavior but cannot classify it into a specific category.

| Subtype | Description | Typical Evidence |
|---|---|---|
| `anomaly_detected` | Statistical anomaly with no pattern match | High anomaly score, no template matched |
| `insufficient_data` | Not enough data to form a hypothesis | Few events, no topology, cold start |

**Diagnostic commands**:
```
docker ps -a
docker logs <container> --tail 500
docker stats --no-stream
dmesg | tail -20
journalctl --since "10 minutes ago"
```

## CauseType Enum

```python
class CauseType(Enum):
    DEPENDENCY_FAILURE = "dependency_failure"
    RESOURCE_EXHAUSTION = "resource_exhaustion"
    APPLICATION_BUG = "application_bug"
    INFRASTRUCTURE_FAILURE = "infrastructure_failure"
    CONFIGURATION_ERROR = "configuration_error"
    DATABASE_FAILURE = "database_failure"
    UNKNOWN = "unknown"
```

## Classification Heuristics

When a hypothesis template generates a hypothesis, it assigns the `CauseType` directly (each template knows its own domain). For cases where the cause type is ambiguous, the following priority applies:

1. If evidence includes deployment/config events before errors â†’ `CONFIGURATION_ERROR`
2. If evidence includes resource metrics at threshold â†’ `RESOURCE_EXHAUSTION`
3. If evidence includes database-specific errors â†’ `DATABASE_FAILURE`
4. If evidence spans multiple services with dependency chain â†’ `DEPENDENCY_FAILURE`
5. If evidence is confined to one service with code-level errors â†’ `APPLICATION_BUG`
6. If evidence includes infrastructure-level signals â†’ `INFRASTRUCTURE_FAILURE`
7. Otherwise â†’ `UNKNOWN`

## Trend Tracking

The taxonomy enables tracking failure patterns over time:

```python
@dataclass
class FailureTrend:
    period: str               # "7d" | "30d"
    cause_type_counts: dict[str, int]    # CauseType â†’ count of incidents
    top_subtypes: list[tuple[str, int]]  # (subtype, count), sorted descending
    top_services: list[tuple[str, int]]  # (service_id, incident_count)
```

Stored in the incident store as a materialized view, updated on incident resolution.

## Extensibility

The taxonomy is intended to cover 90% of failure modes in containerized microservice environments. For domains with specialized failure modes (ML inference, data pipelines, IoT), additional subtypes can be added via configuration:

```toml
[[failure_taxonomy.custom_subtypes]]
cause_type = "application_bug"
subtype = "model_inference_failure"
description = "ML model failed to produce output"
patterns = ["inference error", "model timeout", "prediction failed"]
diagnostic_commands = ["check model server logs", "verify model artifact"]
```

Custom subtypes are merged with built-in subtypes at startup.
